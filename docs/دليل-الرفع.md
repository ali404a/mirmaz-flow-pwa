# دليل الرفع — mirmaz FLOW 2.0

## قبل أي شيء — إجراءات عاجلة

النسخة القديمة سرّبت بيانات فعلياً. نفّذ هذه أولاً:

1. **احذف الـ blob**: كان كل شيء (المهام، كلمات المرور) على
   `jsonblob.com/api/jsonBlob/019f6ab0-...` — رابط عام بلا مصادقة يقبل الكتابة.
   احذفه من jsonblob.com. النسخة 2.0 لا تستخدمه.
2. **بدّل أي كلمة مرور** استُخدمت في النظام القديم (كانت `1234` افتراضياً، بنص صريح).
3. **راجع روابط الدرايف**: 44 رابط Google Drive/Canva كانت مضمّنة في الـHTML العام.
   افتح كل مجلد وتأكد أنه ليس "anyone with the link" إن كان يحوي شيئاً حسّاساً.

## 1) إنشاء مشروع Supabase

1. [supabase.com](https://supabase.com) ← New project
2. اختر منطقة قريبة (Frankfurt أسرع للعراق عادةً)
3. احفظ كلمة مرور القاعدة في مدير كلمات مرور

## 2) تشغيل ملفات القاعدة — بالترتيب

من `SQL Editor > New query`، شغّل بالترتيب:

| # | الملف | ماذا يفعل |
|---|-------|-----------|
| 1 | `database/01_schema.sql` | الجداول والأنواع |
| 2 | `database/02_permissions.sql` | RLS + الحمايات (قابل لإعادة التشغيل) |
| 3 | `database/03_seed.sql` | الأقسام والقنوات فقط |

## 3) إنشاء الحسابات

### أ) علي — السوبر أدمن

`Authentication > Users > Add user`:
- **Email**: `ali@mirmaz.com`
- **Password**: اضغط *Generate a password* (أو `openssl rand -base64 18`)
- ✅ فعّل **Auto Confirm User**

### ب) ياسمين حاكم — الكوردنيتر

نفس الخطوات بـ `yasmin@mirmaz.com`.

### ج) ضبط الأدوار

شغّل `database/04_accounts.sql`. سيجد الحسابين بالبريد ويضبط دوريهما.

> **سلّم كلمات المرور عبر قناة آمنة** — مدير كلمات مرور مشترك، أو شفهياً.
> لا واتساب ولا إيميل عادي.

### د) بقية الفريق

يدخل **علي** بحسابه ← صفحة الفريق ← "إضافة موظف جديد".
كل حساب يُنشأ عبر Edge Function آمنة، ويختار له علي الدور المناسب.

**ملاحظة عن الصلاحيات**: ياسمين (كوردنيتر) تبني المهام والخرائط وتنشر،
لكنها **لا تنشئ حسابات** ولا توافق على المحتوى — الموافقة للإدارة، والحسابات لعلي.
هذا فصل مقصود: من يبني المهمة لا يعتمدها.
إن أردت أن تنشئ ياسمين حسابات، رقّها إلى `manager`.

## 4) نشر Edge Function

```bash
npm i -g supabase
supabase login
supabase link --project-ref <your-project-ref>

mkdir -p supabase/functions/create-user
cp backend/create-user-function.ts supabase/functions/create-user/index.ts
supabase functions deploy create-user
```

ثم في `Dashboard > Edge Functions > create-user > Secrets` أضف:

```
ALLOWED_ORIGIN = https://your-domain.vercel.app
```

> إن تركت `ALLOWED_ORIGIN` فارغاً، الدالة لن تسمح لأي نطاق (فشل مغلق — آمن).
> لا تستخدم `*` في الإنتاج.

## 5) ربط الواجهة

في `frontend/supabase-client.js` بدّل السطرين:

```js
const SUPABASE_URL  = 'https://xxxxx.supabase.co';
const SUPABASE_ANON = 'eyJhbGci...';
```

من `Project Settings > API`.

> **مفتاح anon عام بطبيعته** — آمن للنشر، والحماية الحقيقية في RLS.
> **لا تضع `service_role`** في الواجهة إطلاقاً.

## 6) النشر على Vercel

```bash
npm i -g vercel
vercel --prod
```

`vercel.json` جاهز.

## 7) شاشة العرض (لوحة الوصول السريع)

اللوحة تقرأ من القاعدة، و RLS يمنع القراءة بلا جلسة. لذلك:

1. أنشئ حساباً مخصصاً للعرض (مثلاً `display@mirmaz.com`) بدور `caption_writer`
2. سجّل الدخول به **مرة واحدة** على شاشة الماركتنك
3. الجلسة تبقى، واللوحة تتحدث لحظياً عبر Realtime

> لا تترك شاشة عرض بحساب أدمن — إن مرّ أحد بجانبها يستطيع كل شيء.

## 8) استيراد الأرشيف (اختياري)

`database/05_archive_import.sql` يستورد 31 مهمة حقيقية كانت مضمّنة في الـHTML.
شغّله فقط إن أردت الأرشيف. النظام يعمل بدونه.
